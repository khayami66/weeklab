### セッション5: 運用面・配信形態の確定

**壁打ちした論点**
1. 管理職への事前確認 → **済み**
2. 年度切り替え運用
3. データのバックアップ・端末間移行
4. 教科書改訂への対応
5. 生徒個人情報の扱い

**Webアプリ vs ローカルの比較**
- 配布性・多端末利用・LINE連携・保守性すべてでWebアプリが有利
- Next.js はそもそもWebアプリ前提
- 生徒個人情報を扱わないのでセキュリティリスクは低い

**決定事項**
| 項目 | 決定 |
|---|---|
| 配信形態 | **Webアプリ**（Vercel、無料枠、限定URL） |
| 利用端末 | 自宅PCのみ（学校PCはセキュリティ制約で導入困難） |
| 年度管理 | 年度ごとにデータ分離、年度切替機能をMVPに入れる |
| アーカイブ | 凍結保存（`WeeklyPlan` スナップショット方式） |
| アーカイブ保持期間 | MVPは無期限（多ユーザー化したら見直し） |
| バックアップ | JSONエクスポート/インポートをMVPに追加 |
| 教科書改訂 | 最新パックのみ保持（旧パックは破棄） |
| 過去年度の単元名一貫性 | アーカイブは WeeklyPlan で保存されるため影響なし |
| 生徒個人情報 | 扱わない |
| 成績管理 | 別システムで切り分け、将来API連携で疎結合 |

**データ設計の追加**
- `TeacherSetting.school_year` を追加
- localStorage キーに年度プレフィックス：`weeklab.{year}.*`
- `ArchivedWeek` / `ArchiveMetadata` / `FullSnapshot` 型を追加
- `/archive` 画面を追加
- `/settings` に年度切替モーダル・エクスポート/インポートパネル

### セッション7: 設計レビューと壁打ち

**対応**
- Phase 1 着手前に設計ドキュメントを網羅的に確認
- 潜在的論点を12項目抽出し、順次合意形成

**合意された方針変更**

1. **進度管理：前週確定型に変更**
   - 従来案：`ClassProgress` から全コマ自動計算
   - 新案：**各クラスの週先頭コマをユーザーが確定** → 以降は `LessonMaster` 順で自動表示
   - 理由：実授業は進度がずれるため、週はじめに再確定する方が現実的
   - データ層は変えず、UI層の運用のみ変更
   - キー：`weeklab.{year}.first_lesson.{monday_date}`

2. **進度ヘルスチェック機能を追加**
   - 期待累計 = 年間配当時数 × (現在週 / 全週数)
   - 差 ±3時間で warn（amber）、±5時間で alert（rose）
   - ホーム・週案・進度画面にバッジ表示
   - 新ファイル：`lib/healthCheck.ts`

3. **週案と月次集計を分離**
   - 週案表示・印刷：**週実施・実施累計**のみ（月実施を除外）
   - 月次集計は別画面 `/monthly` に切り出し
   - 印刷対応 `/print/monthly`（教育委員会報告用）
   - 松戸市様式とも整合（元様式も月実施を含まない）

4. **iPad 対応**
   - MVP では試験不要
   - Phase 18 で時間があれば iPad Chrome を触る程度
   - 本格対応はユーザー拡大時

5. **LessonMaster 運用**
   - 「2単元先行整備」：現単元＋次単元を常にカバー
   - 未整備時は `lesson_title = "(未作成)"` でフォールバック、時数カウントは継続

**確定した個別項目**

- プロジェクト名：**weeklab**（Next.js プロジェクト名も統一）
- 始業式日：**2026-04-06**
- クラスコード形式：**`3-1`**（ハイフン区切り、3年×3クラス＋4年×4クラス＝7クラス）
- 月〜土：**最初から6行設定**（土曜授業がない日が多いが、入力がなければ空扱い）
- 時間割：**流動的に変更可能なUI**を用意
- 週あたり授業数：**18コマ**（3年2コマ×3クラス ＋ 4年3コマ×4クラス）
- 週先頭コマ確定の粒度：**各クラスごと**（7クラス分）
